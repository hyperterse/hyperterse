Hyperterse provides a simple command-line interface for running the server, development, and generating artifacts.

## Global flags

These flags are available for all commands:

| Flag        | Short | Description                                            |
| ----------- | ----- | ------------------------------------------------------ |
| `--file`    | `-f`  | Path to configuration file (`.terse` or `.hyperterse`) |
| `--help`    | `-h`  | Show help for any command                              |
| `--version` |       | Show version information                               |

## Commands

### `hyperterse run`

Run the Hyperterse server in production mode.

```bash
hyperterse start -f .hyperterse
# Or simply
hyperterse start -f .hyperterse
```

**Flags:**

| Flag          | Short | Default | Description                                      |
| ------------- | ----- | ------- | ------------------------------------------------ |
| `--port`      | `-p`  | `8080`  | Server port (overrides config file and PORT env) |
| `--log-level` |       | `3`     | Log level: 1=ERROR, 2=WARN, 3=INFO, 4=DEBUG      |
| `--verbose`   | `-v`  |         | Enable verbose logging (sets log level to DEBUG) |

**Examples:**

```bash
# Run on custom port
hyperterse start -f .hyperterse -p 3000

# Run with verbose logging
hyperterse start -f .hyperterse -v

# Run with specific log level
hyperterse start -f .hyperterse --log-level 4
```

**Priority Order:**

Server settings are applied in this order (highest to lowest):

1. CLI flags (`-p`, `--log-level`, `-v`)
2. Config file (`server.port`, `server.log_level`)
3. Environment variables (`PORT`)
4. Defaults (`8080`, `INFO`)

### `hyperterse dev`

Run the server in development mode with hot reload.

```bash
hyperterse start --watch -f .hyperterse
```

The server watches your configuration file and automatically reloads when changes are detected. This is ideal for local development.

**Flags:**

Same as `run` command.

**Features:**

- Watches configuration file for changes
- 500ms debounce delay for rapid saves
- Graceful reload without downtime
- Validates configuration before applying changes

<Aside>
  In development mode, invalid configuration changes are rejected and the
  previous valid configuration remains active.
</Aside>

---

### `hyperterse validate`

Validate a configuration file without starting the server.

```bash
hyperterse validate -f .hyperterse
```

Useful in CI pipelines or pre-deploy checks to ensure your configuration is valid.

---

### `hyperterse init`

Initialize a new Hyperterse configuration file.

```bash
hyperterse init
```

Creates a `.hyperterse` file in the current directory with a basic template.

---

### `hyperterse upgrade`

Upgrade Hyperterse to the latest version.

```bash
hyperterse upgrade
```

Downloads and installs the latest version of Hyperterse.

---

### `hyperterse build`

Build a self-contained deployment directory.

```bash
hyperterse build -f .hyperterse -o dist
```

Creates a self-contained runtime directory in the specified output path. The directory contains the Hyperterse binary, `.hyperterse` root config, and `app/` files (if present). This is the **recommended first step for any deployment**.

**Flags:**

| Flag          | Short | Default | Description                             |
| ------------- | ----- | ------- | --------------------------------------- |
| `--out`       | `-o`  | `dist`  | Output directory                        |
| `--clean-dir` |       | `false` | Clean output directory before building |

**Priority Order:**

Build settings are applied in this order (highest to lowest):

1. CLI flags (`--out/-o`, `--clean-dir`)
2. Config file (`build.out`, `build.clean_dir`)
3. Default (`dist` directory, `clean_dir: false`)

**Output:**

- Self-contained runtime directory at `{output_dir}`
- Contains `hyperterse` binary (or `hyperterse.exe`), `.hyperterse`, and `app/` (if present)
- Ready to run from the output directory

**Examples:**

```bash
# Build to default directory (dist/)
hyperterse build -f .hyperterse

# Build to custom directory
hyperterse build -f .hyperterse -o build

# Clean directory before building
hyperterse build -f .hyperterse -o build --clean-dir

# With config build.out set to "build", CLI flag overrides it
hyperterse build -f .hyperterse -o release
```

<Aside type="caution">
  Always use `hyperterse build` before deploying. The output is
  optimized for production and doesn't require Hyperterse to be
  installed globally.
</Aside>

### `hyperterse generate llms`

Generate an `llms.txt` documentation file for AI/LLM consumption.

```bash
hyperterse generate llms -f .hyperterse
hyperterse generate llms -f .hyperterse -o docs/llms.txt --base-url https://api.example.com
```

**Flags:**

| Flag         | Short | Default                 | Description                |
| ------------ | ----- | ----------------------- | -------------------------- |
| `--output`   | `-o`  | `llms.txt`              | Output file path           |
| `--base-url` |       | `http://localhost:8080` | Base URL for API endpoints |

The generated file contains markdown documentation optimized for AI consumption, including:

- Query descriptions
- Input schemas
- Example requests
- Usage patterns

---

### `hyperterse generate skills`

Generate an Agent Skills archive (`.zip`) for AI platforms.

```bash
hyperterse generate skills -f .hyperterse
hyperterse generate skills -f .hyperterse -o my-skill.zip --name my-data-api
```

**Flags:**

| Flag       | Short | Default                | Description      |
| ---------- | ----- | ---------------------- | ---------------- |
| `--output` | `-o`  | `skill.zip`            | Output file path |
| `--name`   |       | (from config filename) | Skill name       |

The archive contains everything needed to register Hyperterse queries as AI agent skills.

## Environment variables

| Variable | Description                           |
| -------- | ------------------------------------- |
| `PORT`   | Default server port (lowest priority) |

Use environment variables in configuration files:

```yaml
server:
  port: '{{ env.PORT }}'
```

---

## Exit codes

| Code | Meaning                                                  |
| ---- | -------------------------------------------------------- |
| `0`  | Success                                                  |
| `1`  | General error (invalid config, connection failure, etc.) |

---

## Log levels

Control logging verbosity with `--log-level` or `-v`:

| Level | Name  | Description                    |
| ----- | ----- | ------------------------------ |
| `1`   | ERROR | Only errors                    |
| `2`   | WARN  | Warnings and errors            |
| `3`   | INFO  | General information (default)  |
| `4`   | DEBUG | Detailed debugging information |

```bash
# Set specific level
hyperterse start -f .hyperterse --log-level 4

# Shortcut for DEBUG level
hyperterse start -f .hyperterse -v
```
