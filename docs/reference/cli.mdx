## Global behavior

The Hyperterse CLI is built with [Cobra](https://github.com/spf13/cobra). All commands load `.env` files from the working directory if present.

```bash
hyperterse [command] [flags]
```

### Version

```bash
hyperterse --version
```

Prints the binary version and exits.

## `hyperterse start`

Start the runtime from a configuration source. Parses the config, compiles the project, bundles scripts, initializes connectors, and starts the HTTP/MCP server.

```bash
hyperterse start [path] [flags]
```

When run without arguments, the CLI loads `.hyperterse` from the current directory. A positional `path` argument can point to a directory (which must contain `.hyperterse`) or directly to a config file. The `--file` flag is an alternative to the positional argument — they cannot be combined.

### Flags

| Flag | Short | Type | Default | Description |
|---|---|---|---|---|
| `--file` | `-f` | string | `""` | Path to the configuration file (`.hyperterse` or `.terse`). Alternative to positional arg. |
| `--port` | `-p` | string | `""` | Server port (overrides config file and `PORT` env var). |
| `--log-level` | | int | `0` | Log level: 1=ERROR, 2=WARN, 3=INFO, 4=DEBUG (overrides config file). |
| `--verbose` | | bool | `false` | Enable verbose logging (sets log level to DEBUG). |
| `--source` | `-s` | string | `""` | Configuration as a string (alternative to `--file`). |
| `--watch` | | bool | `false` | Watch `.terse`/`.ts` files and hot-reload on changes. |
| `--log-tags` | | string | `""` | Filter logs by tags (comma-separated, use `-tag` to exclude). Overrides `HYPERTERSE_LOG_TAGS` env var. |
| `--log-file` | | bool | `false` | Stream logs to file in `/tmp/.hyperterse/logs/`. |

### Resolution Order

**Port:** CLI `--port` → config `server.port` → env `PORT` → `8080`

**Log level:** `--verbose` (sets 4) → `--log-level` → config `server.log_level` → `3`

### Examples

```bash
hyperterse start
hyperterse start --watch
hyperterse start path/to/project
hyperterse start -p 9090 --verbose
```

## `hyperterse serve`

Start the runtime from a pre-built manifest. Loads `model.bin` and reconstructs the project from the serialized manifest without re-parsing source files.

```bash
hyperterse serve [manifest-or-dir] [flags]
```

When run without arguments, the runtime searches for `model.bin` in the current directory. A positional argument can point to a directory (which must contain `model.bin`) or directly to a manifest file. The `--file` flag is an alternative to the positional argument — they cannot be combined.

### Flags

| Flag | Short | Type | Default | Description |
|---|---|---|---|---|
| `--file` | `-f` | string | `""` | Path to the model manifest file. Alternative to positional arg. Default: `./model.bin`. |
| `--port` | `-p` | string | `""` | Server port (overrides manifest and `PORT` env var). |
| `--log-level` | | int | `0` | Log level: 1=ERROR, 2=WARN, 3=INFO, 4=DEBUG (overrides manifest). |
| `--verbose` | | bool | `false` | Enable verbose logging (sets log level to DEBUG). |
| `--log-tags` | | string | `""` | Filter logs by tags (comma-separated, use `-tag` to exclude). Overrides `HYPERTERSE_LOG_TAGS` env var. |
| `--log-file` | | bool | `false` | Stream logs to file in `/tmp/.hyperterse/logs/`. |

### Examples

```bash
hyperterse serve dist/
hyperterse serve
```

## `hyperterse build`

Compile the project into a deployable output directory. Parses config, compiles the project, bundles all TypeScript scripts, copies the runtime binary, and writes the serialized manifest.

```bash
hyperterse build [path] [flags]
```

When run without arguments, the CLI loads `.hyperterse` from the current directory. A positional `path` argument can point to a directory (which must contain `.hyperterse`) or directly to a config file. The `--file` flag is an alternative to the positional argument — they cannot be combined.

### Flags

| Flag | Short | Type | Default | Description |
|---|---|---|---|---|
| `--file` | `-f` | string | `""` | Path to the configuration file (`.hyperterse` or `.terse`). Alternative to positional arg. |
| `--out` | `-o` | string | `""` | Output directory for build artifacts. Default: `dist`. |
| `--clean-dir` | | bool | `false` | Remove existing output directory before building. |

### Output Structure

```
dist/
├── hyperterse          # Runtime binary
├── model.bin           # Serialized manifest
└── build/
    ├── vendor.js       # Shared dependency bundle
    └── routes/
        └── <name>/
            ├── handler.js
            ├── input_transform.js
            └── output_transform.js
```

### Examples

```bash
hyperterse build
hyperterse build -o release --clean-dir
hyperterse build path/to/project -o dist
```

## `hyperterse validate`

Validate the configuration and project structure without starting the runtime. Checks config schema compliance, adapter definitions, route validity, script resolution, and bundling feasibility.

```bash
hyperterse validate [path] [flags]
```

When run without arguments, the CLI loads `.hyperterse` from the current directory. A positional `path` argument can point to a directory (which must contain `.hyperterse`) or directly to a config file. The `--file` flag is an alternative to the positional argument — they cannot be combined.

### Flags

| Flag | Short | Type | Default | Description |
|---|---|---|---|---|
| `--file` | `-f` | string | `""` | Path to the configuration file (`.hyperterse` or `.terse`). Alternative to positional arg. |
| `--source` | `-s` | string | `""` | Configuration as a string (alternative to `--file`). |

### Validation Checks

- Root config schema compliance (name, version, server, build fields).
- Adapter completeness (connector and connection_string required).
- Route validity (each route has `use` or `scripts.handler`).
- Adapter name uniqueness.
- Input type correctness (valid primitive types).
- Script file resolution (referenced files exist).
- Bundle compilation (scripts compile without errors).

### Exit Codes

| Code | Meaning |
|---|---|
| `0` | Validation passed. |
| `1` | Validation failed. Errors are printed to stderr. |

### Examples

```bash
hyperterse validate
hyperterse validate path/to/project
```

## `hyperterse init`

Scaffold a new Hyperterse project with starter files.

```bash
hyperterse init
```

### Created Files

| Path | Content |
|---|---|
| `.hyperterse` | Root config with default name and server settings. |
| `app/adapters/my-database.terse` | Placeholder adapter definition. |
| `app/routes/health/config.terse` | Health check route configuration. |
| `app/routes/health/handler.ts` | Health check handler script. |

### Behavior

- Creates files in the current working directory.
- Does not overwrite existing files.

## `hyperterse upgrade`

Upgrade the installed Hyperterse binary to a newer version.

```bash
hyperterse upgrade [flags]
```

### Flags

| Flag | Type | Default | Description |
|---|---|---|---|
| `--prerelease` | bool | `false` | Include pre-release versions in upgrade candidates. |
| `--major` | string | | Target a specific major version number or `next` for the next major. |

### Examples

```bash
hyperterse upgrade
hyperterse upgrade --prerelease
hyperterse upgrade --major next
hyperterse upgrade --major 3
```

## `hyperterse completion`

Generate shell completion scripts. This command is hidden from the default help output and is primarily used by the installer.

```bash
hyperterse completion [bash|zsh|fish|powershell]
```
